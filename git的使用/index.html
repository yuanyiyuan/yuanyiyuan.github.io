Git的使用

1.首先，什么是Git
Git是目前世界上最先进的【分布式版本控制系统】

    那为什么要出现Git?出现解决的问题是什么呢？

    假设有一篇长篇论文，想要删除一个段落，又怕文档删除后找不回来，只好先保存文件，另存为xxx，之后再次修改，修改后又另存为xxx,老师看到不满意，
    再次修改，然后修改后又另存为，（然后就是一堆另存为文件，最后自己也找不到所需要的文档了，然后一个一个去查找，浪费时间和精力），若之后有人帮忙修改，修改后又要另存为，然后你想看她改了哪里，然后这又是困难。

<<<但若有一个软件，不但可以自动记录数据的改动，还可以多人编辑，同时记录编辑了那些地方>>—>多么方便—>Git的出现

2.Git的分布式管理 与 SVN集中式管理 对比
集中式管理：版本库集中存放在中央服务器，干活用的是自己的电脑
—–使用时先用自己的电脑去中央服务器取得最新的版本，然后干活，干完后再把文件推送到中央服务器。（必须要联网才能down、工作）。只要中央服务器有完整的版本库
分布式管理：没有了“中央服务器”，每个人的电脑都是一个完整的版本库，这样工作就不用去down、联网。 同时安全性提高，因为每个人电脑里都有完整的版本库
—–正常时候可能大家都不在一个局域网中，两个电脑无法相互访问，分布式版本通常也有一个电脑充当“中央服务器”–这个大家都知道，叫Github，(仅仅是解决大家相互交换修改不方便而已)
<<同时还有强大的分支管理>>

3.Git大门的正式步入
3.1 安装Git
|
|—>Linux : 终端输入： git 【来检测是否安装Git】 sudo apt-get install git 【用来安装Git】
|
|—>Windows : 安装完成后，在菜单里找到”Git”->”Git Bash”,弹出一个命令行的窗口，说明安装成功
| git config –global user.name “your name”
| git config –global user.email “your email”
| 用了global,此机器上的所以Git仓库都会使用这个配置 ，【姓名和邮箱是方便分布式管理及查询】

3.2 创建版本库
    是不是又人要问了，什么是版本库，前面一直提版本库，他到底是什么？
    版本库就是我们说的仓库(repositiry), 用于存放文件的, 可以理解成一个目录，Git管理此目录里的所有文件，Git可以记录,跟踪每个文件的修改、删除, 可以任意时刻去追踪历史，还原某个时刻的文件

    是不是特别神奇，我们来一起来进军Git

    创建版本库： 【我的目录叫learngit ,可随意取名】
    1.首先选择一个合适的地方，创建一个空目录，不要是中文路径
       mkdir learngit      【创建一个空目录】
       cd learngit         【进入创建的目录】
    2.把这个目录变成一个Git管理的仓库
       git init    【变成了一个Git管理的仓库】--Git只能管理对应目录下的所有文件
    
    添加文件到Git仓库：   【我的文件叫readme.txt ,可随意取名】
    1.把文件添加到仓库
       git add readme.txt
    2.把文件提交到仓库(可以一次提交多个文件)    【-m后 为本次提交的说明，可为任意内容，最好有意义，方便之后查找】
       git commit -m "wrote a readme file"

------------------------------------------------------
内部结构：

|工作区：   |           |版本库：
|          |     add    |  缓存区：       提交区：master
| 写下文件 |  --------> |  第一次添        最后提交的
| 的地方   |            |  加的地方 -----> 本地的地方
|          |            |
|          |            |
------------------------------------------------------
Git 是跟踪修改的，每次修改，如果不add到缓存区，那就不会加到提交区
------------------------------------------------------

3.3时光穿越机（常用Git的地方）
查看工作区的Git状态：   git status      【当提示working directory clean , 则为都提交成功（其实都是就是看缓存区是否都提交到master里）】
查看两次文档之间有什么修改：   git diff  readme.txt （文件名）
查看提交的Git命令：   git log    【以便回退到对应的版本】
回退版本的Git命令：   git reset  --hard  HEAD^   (HEAD^ 上一个版本   HEAD^^ 上上一个版本  HEAD~100  往上的100个版本)
    回退版本后，突然回不去了，从21世纪穿越到19世纪，然后回不去了，哈哈哈，这个是不可能发生的，我们用 git log 看过版本，里面有每个版本对应的
    id,我们可以通过id找到对应的时空，eg:--  git reset --hard 3628164(这个为id号，输入前面的就会自动查找)<回退速度非常快，因为Git内部有一个HEAD指针，
    Git仅仅是把指针指向了对应的版本>
查看Git命令历史：   git reflog
当扰乱了工作区的某个文件，想直接丢弃工作区的修改，用Git命令来丢弃： git checkout --readme.txt （文件名）
当扰乱了工作区的某个文件,还提交到了缓存区，想丢弃修改，用Git命令来丢弃： git reset HEAD readme.txt （文件名）  【回到当前文件】
                                                                    git checkout --readme.txt （文件名）
删除文件：先删除文件： rm readme.txt （文件名） 
         后删除版本库的文件： git rm readme.txt （文件名）
若误删后，可用 git checkout -- text.txt  恢复到最新的版本

好啦好啦，为什么用git checkout -- text.txt这个指令呢，这个指令为， 用版本库的版本替换工作区的版本，无论工作区是修改或删除，都可一键恢复

有了这个方法，我们就可以穿越各个版本，去放心大胆的尝试吧

  1. 建立远程的仓库来备份
    github 的注册和登录
    我们要的是推送： git push origin master 【这样就推送到了Github远程仓库上了】
    克隆仓库： git clone 仓库地址

    Git为一个分布式的版本库，可以分支，因为创建、合并、删除分支特别快，所以可以分支完成某个任务，然后合并再删除分支，这个和直接在master分支上工作效果一样，但过程安全
    给举个例子，若我有一个任务，要分3个部分完成，这样我就可以分为3个分支，每完成一个后，没问题后再合并后删除分支，这样有条理，还方便

    查看分支： git branch
    创建分支： git branch name
    切换分支： git checkout name
    创建+切换分支： git checkout -b name
    合并某个分支到当前分支： git merge name
    删除分支： git branch -d name
    查看分支合并图： git log –graph

    <合并分支时，加上--no-ff，就可以用普通模式合并，合并后的历史有分支，有痕迹，而默认合并为 fast forward 合并就看不出来曾做过合并>

    若突然出现问题，要把当前任务停一下，但还不想提交，可以进入BUG模式：

     git stash     【把工作先存到临时存储区】  用 git status 查看工作区，应该是干净的了
 git checkout 分支   【跳转到对应的分支上来修复，建立临时分支来修复文件】
 git checkout 当前分支 【完成后跳转到当前的分支继续完成自己的工作】
 git stash list  【查看文件存储到存储区某个位置】
 git stash apply 存储区  【文件恢复，但存储区不删除，要再 git stash drop 来删除】 或者 【用 git stash pop  存储区 恢复的同时把stash的内容给删除】

    开发一个新的功能的时候，最好新建一个分支，若要丢弃一个没有合并的分支，可以 git branch -D name 强行删除
    查看远程库信息： git remote -v
    从本地推送分支： git push origin branch-name
    在本地创建和远程分支对应的分支： git checkout -b branch-name origin/branch-name
    建立本地分支和远程分支的关联： git brach –set-upstream branch-name origin/branch-name
    从远程抓取分支： git pull
